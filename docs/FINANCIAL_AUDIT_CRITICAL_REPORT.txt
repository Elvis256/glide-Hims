## 1. DOUBLE-ENTRY INTEGRITY (IFRS/IAS)
**Rating: 6/10** - Minimal validation, structural weakness

### GAPS IDENTIFIED:

✗ **VALIDATION WEAKNESS**
  - Location: finance.service.ts:214
  - Code: `if (Math.abs(totalDebit - totalCredit) > 0.01) throw`
  - Issue: Only 0.01 precision tolerance. Floating-point arithmetic can produce
    rounding errors. No explicit review queue for edge cases.
  - Example: 3 transactions totaling 1000.00 could round to 1000.01, pass validation,
    but not actually balance. This violates fundamental accounting principle.

✗ **NO PRE-POSTING VERIFICATION QUEUE**
  - Location: Missing from finance.controller.ts
  - Issue: Journal entries transition from DRAFT → POSTED with ONE step.
    In IFRS, entries should have:
    1. DRAFT (created)
    2. READY_FOR_POSTING (balance verified)
    3. POSTED (approved & locked)
  - Current: postJournalEntry() at line 301 skips intermediate step.
  - Risk: Unbalanced entries could be posted if validation tolerance is exploited.

✗ **NO BALANCE NOTIFICATION TO USERS**
  - Location: finance.controller.ts (POST /journals/:id/post)
  - Issue: User posts entry without seeing verification. If debit≠credit, error 
    thrown but user has NO way to see "account X is short 50,000"
  - Impact: User confusion, frustration, re-entry without understanding root cause.

✓ Good: createJournalEntry() validates debit=credit before saving (line 211-216)

### RECOMMENDATION:
- Use DECIMAL(15,2) consistently, never float
- Implement 2-phase posting: verify balance, THEN post
- Return detailed balance report when posting

## 3. SEPARATION OF DUTIES (SOX-LIKE)
**Rating: 2/10** - SEVERELY DEFICIENT, Fraud enablement risk

### CRITICAL GAPS:

✗ **NO MAKER-CHECKER ON JOURNAL ENTRIES**
  - Issue: Same user can createJournalEntry() AND postJournalEntry()
  - Locations:
    - Create: finance.controller.ts:126, passes req.user.id to create()
    - Post: finance.controller.ts:157, passes req.user.id to post()
  - No role separation enforcement
  - A single user could:
    1. Create JE: DR Bad Debt Expense, CR Revenue (fraud)
    2. Post JE immediately (no approval)
    3. Modify their own GL balances
  - Risk: HIGH - no second pair of eyes on fraudulent entries

✗ **NO APPROVAL WORKFLOW FOR PAYMENTS**
  - Issue: recordPayment() in billing.service.ts:266 is IMMEDIATE
  - Locations:
    - billing.controller.ts:117-122: POST /payments goes directly to service
  - Expected flow: CREATE → PENDING → APPROVED → PAID
  - Current flow: CREATE → PAID (1 step)
  - A cashier can:
    1. Record fake payment
    2. Invoice balance due becomes 0
    3. Patient never actually paid
  - No approval from supervisor/manager

✗ **MISSING DUAL CONTROL ON INVOICES**
  - Issue: Same user who creates can also cancel/refund
  - Locations:
    - Create: billing.controller.ts:23, user creates
    - Cancel: billing.controller.ts:69, same user can cancel
    - Refund: billing.controller.ts:80, same user can refund
  - Expected: CREATE by cashier, CANCEL/REFUND by finance manager
  - Current: Single user can do both
  - Risk: Patient refunded without authorization, money diverted

✗ **WRITE-OFFS NOT SEGREGATED**
  - Location: billing.controller.ts:157-166
  - writeOffInvoice() requires 'finance.manage' permission
  - But NO separate approval from a different user
  - A finance manager can:
    1. Create fake invoice
    2. Immediately write it off as bad debt
    3. Expense recorded, no detection

✗ **NO AUTHORIZATION MATRIX**
  - Issue: No role-based GL account posting restrictions
  - Example: User with 'finance.manage' can post to ANY account
    - Could post to REVENUE (inflate revenue)
    - Could post to EXPENSE (inflate expenses)
    - Could post to CASH (hide fraud)
  - No restriction on which accounts a user can touch

✗ **VOIDPAYMENT() HAS NO APPROVAL**
  - Location: billing.controller.ts:124-133
  - voidPayment() is IMMEDIATE, requires only 'billing.update'
  - No secondary approval
  - A cashier can:
    1. Record payment
    2. Immediately void it
    3. Patient no longer owes
    4. No GL adjustment

✗ **NO SEGREGATION FOR SUPPLIER PAYMENTS**
  - Location: supplier-finance.service.ts:167-176
  - approvePaymentVoucher() is a separate step, GOOD
  - BUT processPayment() can update cheque numbers AFTER approval (line 187-192)
  - This allows: Approve payment for supplier A, then change cheque to supplier B

### CURRENT PERMISSION MODEL:
From code review:
- 'billing.create' - can create invoices
- 'billing.update' - can modify/void ANY invoice
- 'finance.manage' - can post to GL, write off
- No separation between "invoice creator" vs "invoice approver"
- No segregation of accounts (can't restrict which GL accounts user touches)

### RECOMMENDATION - IMPLEMENT IMMEDIATELY:
```
Maker-Checker roles:
  - OPERATOR: creates invoices/payments (cannot post/approve)
  - REVIEWER: approves invoices/payments
  - POSTED: immutable once approved
  
GL Authorization Matrix:
  - By account code, not just generic permission
  - Revenue accounts: Only revenue manager
  - Expense accounts: Only department manager
  - Cash accounts: Only treasurer
  - AR/AP accounts: Only finance controller
```

## 4. REVENUE RECOGNITION
**Rating: 3/10** - Incorrect, non-compliant, no flexibility

### CRITICAL FLAWS:

✗ **REVENUE RECOGNIZED AT INVOICE CREATION (WRONG)**
  - Location: billing.service.ts:118-129
  - Current: autoPostInvoiceJournal() called when createInvoice()
  - Code: DR Accounts Receivable, CR Revenue (immediately)
  - IFRS Issue: Revenue should be recognized when:
    1. Performance obligation satisfied (service delivered), NOT invoice created
    2. Patient payment received (cash basis for healthcare)
    3. Insurance pre-auth verified (for insurance patients)
  - Current: A patient is billed for a 3-month treatment plan, all revenue 
    recognized on day 1, before treatment starts or is delivered
  - Impact: Revenue overstated, financial statements IFRS non-compliant

✗ **NO DEFERRED REVENUE ACCOUNT**
  - Expected: For services paid upfront or insurance pre-payments
  - Missing: No GL mapping for deferred revenue
  - Finance service has no pre-payment handling
  - Result: If patient deposits 1M now for future visits, ALL recorded as revenue today

✗ **NO ACCRUAL/PREPAYMENT SUPPORT**
  - Missing: Not in finance.service.ts
  - Example: Patient receives IV bag worth 500K on credit:
    1. Should create: DR Service Expense, CR Accounts Payable
    2. Later when paid: DR Accounts Payable, CR Cash
  - Current: Only ONE entry is possible (at invoice creation)
  - Result: Can't track the expense timing correctly

✗ **REVENUE NOT CONFIGURABLE**
  - Location: pricing-engine.service.ts:561-602
  - autoPostInvoiceJournal() hardcodes revenue category mapping:
    - 'consultation' → account 4100
    - 'lab' → account 4102
    - etc.
  - Issue: These account mappings are NOT configurable per facility
  - If facility uses different chart of accounts, entries go to WRONG accounts
  - Result: GL balances meaningless, reports incorrect

✗ **NO REVENUE CUTOFF CONTROL**
  - Issue: No procedure to verify revenue for period end vs actual service dates
  - If invoice created on Jan 31 for Feb service, it's recorded as Jan revenue
  - Expected: Cutoff test (revenue matched to service date, not invoice date)

### CODE EXAMPLE OF THE PROBLEM:
```typescript
// billing.service.ts:118-129

  this.financeService.autoPostInvoiceJournal({
    // ...
    revenueCategory: dto.paymentType || 'consultation', // May be WRONG category
  })

// But encounterId might be updated later, or service might not start yet
// Revenue already booked!
```

### RECOMMENDATION:
- Separate invoice creation from revenue recognition
- Add revenue recognition event: 
  - For cash: when payment received
  - For insurance: when claim approved + verified
  - For deferred: when service delivery date
- Implement deferred revenue GL accounts
- Add period-end cutoff review step

## 6. INVOICE CONTROLS
**Rating: 4/10** - Gaps in sequencing and deletion

### CRITICAL ISSUES:

✗ **INVOICE NUMBERS NOT TRULY SEQUENTIAL & GAPPABLE**
  - Location: billing.service.ts:31-53 (generateInvoiceNumber)
  - Current logic:
    ```typescript
    const qb = this.invoiceRepository
      .createQueryBuilder('inv')
      .where('inv.invoice_number LIKE :prefix', { prefix: `INV${datePrefix}%` })
      .orderBy('inv.invoice_number', 'DESC');
    const last = await qb.getOne();
    let sequence = 1;
    if (last) {
      const lastSeq = parseInt(last.invoiceNumber.slice(-4), 10);
      sequence = lastSeq + 1;
    }
    return `INV${datePrefix}${sequence.toString().padStart(4, '0')}`;
    ```
  - Issues:
    1. **RACE CONDITION**: Two simultaneous requests get same invoice number
       - T1: Reads last invoice = INV20240101-0050
       - T2: Reads last invoice = INV20240101-0050 (same!)
       - Both generate INV20240101-0051
       - Database UNIQUE constraint saves us, but second request fails
       - No graceful handling, user gets error instead of next available
    2. **Resets daily**: INV20240101-0001...9999, then INV20240102-0001
       - Not auditor-friendly (gaps in yearly sequence)
       - Tax authority wants continuous annual numbering
    3. **Can't skip sequences**: No mechanism to reserve or invalidate numbers
       - If INV20240101-0051 created in error, can't reuse that number
    4. **No transaction-level lock**: Could generate duplicate if DB fails mid-insert

✗ **NO INVOICE DELETION CONTROL**
  - Location: No API endpoint to DELETE invoice (good!)
  - BUT: Entity has no soft_deleted column
  - Issue: If somehow deleted, no audit trail
  - Expected: Invoices should be VOIDED not deleted, CANCELLED not removed
  - Current: Has CANCELLED status, but entity could be deleted from DB

✗ **CREDIT NOTES NOT PROPERLY LINKED**
  - Issue: PatientCreditNote (finance-extended.entity.ts:220-257) has invoiceId
  - BUT: No reverse GL entry when credit note applied
  - Current: Credit note created, but GL shows:
    - Original invoice: DR AR, CR Revenue (still there)
    - Credit note: Applied to invoice? But no DR Revenue reversal
  - Result: GL shows both revenue and credit, double-counting

✗ **NO CANCELLATION REVERSAL**
  - Location: cancelInvoice() at billing.service.ts:532-554
  - Code: Sets status = CANCELLED, NO GL reversal
  - Example:
    1. Invoice created: DR AR 1M, CR Revenue 1M
    2. Invoice cancelled: (nothing in GL)
    3. GL still shows 1M revenue, but invoice cancelled
    - Financial statements WRONG

✗ **VOIDED PAYMENTS NOT REVERSED IN GL**
  - Location: voidPayment() at billing.service.ts:429-465
  - Code: Sets payment status = VOIDED, updates invoice balance
  - BUT: autoPostPatientPaymentJournal() was called when payment recorded
    - DR Cash, CR AR (posted)
    - Void payment? No GL reversal
  - Result: GL shows cash receipt that was voided

✗ **REFUND HANDLING**

  - Code: Sets status = REFUNDED, no GL entry
  - Current: Invoice shows REFUNDED, but GL is unchanged
  - Expected: GL should have DR Refund Payable (liability), CR AR

✗ **INVOICE ITEMS CAN BE ADDED AFTER PAYMENT**
  - Location: addItem() at billing.service.ts:225-243
  - Code: Only checks invoice status ≠ PAID
  - Issue: Invoice status is PARTIALLY_PAID, user adds item

## 8. TAX COMPLIANCE (Uganda URA)
**Rating: 2/10** - EXTREMELY DEFICIENT, Regulatory risk
