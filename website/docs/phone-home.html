Phone Home

Periodic heartbeat from on-premise deployments to the control plane

Overview — The phone-home system sends a periodic heartbeat from each on-premise or hybrid deployment back to the central Glide HIMS control plane. This enables license validation, health monitoring, update checks, and usage telemetry — all without requiring inbound network access to the customer's infrastructure.
How It Works

On a configurable interval (default: every 6 hours), the on-premise instance sends an HTTPS POST request to the control plane with:

  • License key — so the control plane can verify the deployment is still licensed
  • Hardware ID — a fingerprint of the server hardware (CPU, MAC address, etc.)
  • Current version — which version of Glide HIMS is running
  • System metrics — CPU, memory, and disk utilization
  • Active user count — how many users logged in during the interval

The control plane responds with:

  • License status — whether the license is still valid, approaching expiry, or revoked
  • Available updates — newer versions the deployment can upgrade to
  • Feature flags — any remotely-toggled feature flags for this tenant
  • Poll request — whether the control plane wants an immediate health report
API Endpoint
POST /api/v1/phone-home

Public endpoint — authenticated by the license key in the request body.

Configuration

Phone-home behaviour is controlled by environment variables on the on-premise deployment:

VariableDefaultDescription
PHONE_HOME_URL https://hmisdemo.itsolutionsuganda.com Base URL of the Glide HIMS control plane. The heartbeat is sent to {PHONE_HOME_URL}/api/v1/phone-home.
PHONE_HOME_INTERVAL 21600 (6 hours) Interval in seconds between heartbeats. Minimum recommended: 3600 (1 hour).
PHONE_HOME_ENABLED true Set to false to completely disable phone-home (see Air-Gapped Deployments below).
LICENSE_KEY — The license key sent with each heartbeat for validation.
Example .env configuration
# Phone-home settings
PHONE_HOME_URL=https://hmisdemo.itsolutionsuganda.com
PHONE_HOME_INTERVAL=21600
PHONE_HOME_ENABLED=true
LICENSE_KEY=GLIDE-A1B2-C3D4-E5F6-G7H8
License Validation

Each phone-home heartbeat doubles as a license validation check. The control plane:

  1. Looks up the license by key in its database
  2. Verifies the cryptographic signature has not been tampered with
  3. Checks expiration date and revocation status
  4. Optionally verifies hardware binding (if the license is hardware-locked)
  5. Returns the validation result in the response

If the license is expired or revoked, the on-premise deployment enters a restricted mode: existing data remains accessible in read-only mode, but new records cannot be created until a valid license is installed.

Grace Period — If the control plane is unreachable, the deployment continues to operate normally for up to 30 days using the last cached validation result. After 30 days without a successful phone-home, the system enters restricted mode.
Update Checks

The phone-home response includes information about available updates. When an update is available, the on-premise deployment can:

  • Display a notification to system administrators
  • Download the update package via the provided downloadUrl
  • Apply the update automatically (if configured) or wait for manual approval

Updates are distributed through the Updates & Rollouts system, which supports canary, staged, and immediate rollout strategies.

Air-Gapped Deployments

For deployments in environments without internet access (government facilities, military installations, remote clinics), phone-home can be completely disabled:

# Disable phone-home for air-gapped environments
PHONE_HOME_ENABLED=false

When phone-home is disabled:

  • License validation — uses an offline signature file (/etc/glide-hims/license.sig) bundled in the installer bootstrap. The signature is verified locally using the embedded public key.
  • Updates — must be applied manually via USB drive or local network transfer. See Offline Updates.
  • Feature flags — defaults are compiled into the build. Remote toggling is unavailable.
  • Monitoring — health data is only available locally via the admin dashboard. No telemetry is sent externally.
Tip — For air-gapped deployments, ensure the OFFLINE_LICENSE_SIGNATURE_FILE environment variable points to the license signature file. The bootstrap installer sets this up automatically when you provision a standalone deployment.
Security & Privacy
  • All phone-home traffic uses HTTPS/TLS 1.2+
  • No patient data (PII/PHI) is ever transmitted — only aggregate metrics
  • The license key authenticates each request; no additional credentials are needed
  • Hardware IDs are one-way hashes — the actual hardware details cannot be recovered
  • Phone-home can be fully disabled with a single environment variable
Troubleshooting
SymptomCauseSolution
"Phone home failed" in logs Control plane unreachable Check network connectivity and PHONE_HOME_URL. Verify DNS resolution and firewall rules allow outbound HTTPS.
License shows as invalid Expired or revoked key Check the license status on the control plane admin panel. Renew or issue a new key.
Restricted mode after 30 days No successful phone-home for 30 days Restore network access or switch to air-gapped mode with an offline license signature.
Hardware mismatch error Server hardware changed (VM migration, new NIC) Re-bind the license to the new hardware via the admin API or contact support.
