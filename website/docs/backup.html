Backup & Recovery

Protect your healthcare data with robust backup strategies

✓ Verified in v1.0 — production deployments ship with scripts/backup.sh running nightly via cron (encrypted PostgreSQL dump + uploads tarball). Restore is performed with scripts/restore.sh <archive>. The Compliance Centre at /system/compliance tracks Backup/Restore evidence per environment for auditor sign-off.
Critical: Healthcare data is sensitive and often legally protected. Ensure your backup strategy complies with local regulations (HIPAA, GDPR, etc.) and includes encryption.

Backup Type Frequency Retention Storage
Database - Full Daily at 2:00 AM 30 days Off-site / Cloud

Database Backup

Manual Database Backup 
# Full database backup
pg_dump -h localhost -U glide -d glide_hims -F c -f backup_$(date +%Y%m%d_%H%M%S).dump

# With compression
pg_dump -h localhost -U glide -d glide_hims | gzip > backup_$(date +%Y%m%d).sql.gz

# Specific tables only (e.g., for testing)
pg_dump -h localhost -U glide -d glide_hims -t patients -t appointments > partial_backup.sql

# Schema only (no data)
pg_dump -h localhost -U glide -d glide_hims --schema-only > schema_backup.sql
Automated Backup Script 
#!/bin/bash
# /opt/glide-hims/scripts/backup-database.sh

set -e

# Configuration
DB_HOST="localhost"
DB_USER="glide"
DB_NAME="glide_hims"
BACKUP_DIR="/var/backups/glide-hims/database"
S3_BUCKET="s3://your-backup-bucket/glide-hims/database"
RETENTION_DAYS=30
DATE=$(date +%Y%m%d_%H%M%S)
BACKUP_FILE="glide_hims_${DATE}.dump"

# Create backup directory
mkdir -p $BACKUP_DIR

# Create backup
echo "[$(date)] Starting database backup..."
pg_dump -h $DB_HOST -U $DB_USER -d $DB_NAME -F c -f "$BACKUP_DIR/$BACKUP_FILE"

# Compress backup
gzip "$BACKUP_DIR/$BACKUP_FILE"
BACKUP_FILE="${BACKUP_FILE}.gz"

# Calculate checksum
sha256sum "$BACKUP_DIR/$BACKUP_FILE" > "$BACKUP_DIR/${BACKUP_FILE}.sha256"

# Upload to S3 (optional)
if command -v aws &> /dev/null; then
    echo "[$(date)] Uploading to S3..."
    aws s3 cp "$BACKUP_DIR/$BACKUP_FILE" "$S3_BUCKET/$BACKUP_FILE"
    aws s3 cp "$BACKUP_DIR/${BACKUP_FILE}.sha256" "$S3_BUCKET/${BACKUP_FILE}.sha256"
fi

# Clean up old backups
echo "[$(date)] Cleaning up old backups..."
find $BACKUP_DIR -name "*.dump.gz" -mtime +$RETENTION_DAYS -delete
find $BACKUP_DIR -name "*.sha256" -mtime +$RETENTION_DAYS -delete

# Report
BACKUP_SIZE=$(du -h "$BACKUP_DIR/$BACKUP_FILE" | cut -f1)
echo "[$(date)] Backup completed: $BACKUP_FILE ($BACKUP_SIZE)"

File Uploads Backup

File Backup Script 
#!/bin/bash
# /opt/glide-hims/scripts/backup-files.sh

set -e

# Configuration
UPLOADS_DIR="/root/glide-Hims/packages/backend/uploads"
BACKUP_DIR="/var/backups/glide-hims/files"
S3_BUCKET="s3://your-backup-bucket/glide-hims/files"
DATE=$(date +%Y%m%d)

# Create backup directory
mkdir -p $BACKUP_DIR

echo "[$(date)] Starting file backup..."

# Create incremental backup using rsync
rsync -av --delete \
    --backup --backup-dir="$BACKUP_DIR/incremental_$DATE" \
    "$UPLOADS_DIR/" "$BACKUP_DIR/current/"

# Create compressed archive weekly
if [ "$(date +%u)" -eq 7 ]; then
    echo "[$(date)] Creating weekly archive..."
    tar -czf "$BACKUP_DIR/files_weekly_$DATE.tar.gz" -C "$BACKUP_DIR" current/
    
    # Upload to S3
    if command -v aws &> /dev/null; then
        aws s3 cp "$BACKUP_DIR/files_weekly_$DATE.tar.gz" "$S3_BUCKET/"
    fi
fi

# Sync to S3 (daily)
if command -v aws &> /dev/null; then
    echo "[$(date)] Syncing to S3..."
    aws s3 sync "$BACKUP_DIR/current/" "$S3_BUCKET/current/" --delete
fi

echo "[$(date)] File backup completed"

Docker Backup

#!/bin/bash
# Backup Docker volumes and database

DATE=$(date +%Y%m%d_%H%M%S)
BACKUP_DIR="/var/backups/glide-hims"

# Stop services for consistent backup
docker compose stop backend frontend

# Backup PostgreSQL data volume
docker run --rm \
    -v glide-hims_postgres_data:/data \
    -v $BACKUP_DIR:/backup \
    alpine tar czf /backup/postgres_data_$DATE.tar.gz -C /data .

# Backup Redis data volume
docker run --rm \
    -v glide-hims_redis_data:/data \
    -v $BACKUP_DIR:/backup \
    alpine tar czf /backup/redis_data_$DATE.tar.gz -C /data .

# Backup uploads volume
docker run --rm \
    -v glide-hims_uploads:/data \
    -v $BACKUP_DIR:/backup \
    alpine tar czf /backup/uploads_$DATE.tar.gz -C /data .

# Restart services
docker compose start backend frontend

# Alternative: Database dump while running
docker compose exec -T postgres pg_dump -U glide glide_hims | gzip > $BACKUP_DIR/database_$DATE.sql.gz

echo "Docker backup completed: $BACKUP_DIR"

Cloud Provider Backups

AWS RDS
# Enable automated backups
aws rds modify-db-instance \
  --db-instance-identifier glide-hims-db \
  --backup-retention-period 30 \
  --preferred-backup-window "02:00-03:00"

# Create manual snapshot
aws rds create-db-snapshot \
  --db-instance-identifier glide-hims-db \
  --db-snapshot-identifier glide-hims-$(date +%Y%m%d)
Google Cloud SQL
# Enable automated backups
gcloud sql instances patch glide-hims-db \
  --backup-start-time=02:00 \
  --retained-backups-count=30

# Create on-demand backup
gcloud sql backups create \
  --instance=glide-hims-db \
  --description="Manual backup"
Azure Database
# Configure backup retention
az postgres flexible-server update \
  --resource-group glide-hims-rg \
  --name glide-hims-db \
  --backup-retention 30

# Geo-redundant backup
az postgres flexible-server update \
  --name glide-hims-db \
  --geo-redundant-backup Enabled

Recovery Procedures

Warning: Always test recovery procedures in a non-production environment first. Restoring a backup will overwrite existing data.
Database Recovery
Step 1: Stop the Application
# PM2
pm2 stop glide-hims-backend

# Docker
docker compose stop backend
Step 2: Download Backup (if from S3)
aws s3 cp s3://your-backup-bucket/glide-hims/database/glide_hims_20260410.dump.gz /tmp/
Step 3: Restore Database
# Decompress if needed
gunzip /tmp/glide_hims_20260410.dump.gz

# Drop existing database (CAUTION!)
psql -U postgres -c "DROP DATABASE IF EXISTS glide_hims;"
psql -U postgres -c "CREATE DATABASE glide_hims OWNER glide;"

# Restore from dump
pg_restore -h localhost -U glide -d glide_hims -F c /tmp/glide_hims_20260410.dump

# Or from SQL file
psql -h localhost -U glide -d glide_hims < /tmp/backup.sql
Step 4: Verify and Restart
# Verify data
psql -U glide -d glide_hims -c "SELECT COUNT(*) FROM patients;"

# Restart application
pm2 start glide-hims-backend

# Check logs
pm2 logs glide-hims-backend --lines 50
File Recovery 
# Restore from S3
aws s3 sync s3://your-backup-bucket/glide-hims/files/current/ /root/glide-Hims/packages/backend/uploads/

# Restore from tar archive
tar -xzf /var/backups/glide-hims/files/files_weekly_20260407.tar.gz -C /root/glide-Hims/packages/backend/

# Restore specific directory
tar -xzf backup.tar.gz -C /destination/path --strip-components=1 uploads/patients/
Point-in-Time Recovery (PITR)

For critical production systems, enable PostgreSQL WAL archiving for point-in-time recovery:

# postgresql.conf
wal_level = replica
archive_mode = on
archive_command = 'aws s3 cp %p s3://your-bucket/wal/%f'

# Recover to specific time
pg_restore --target-time="2026-04-10 14:30:00" ...

Backup Verification

#!/bin/bash
# /opt/glide-hims/scripts/verify-backup.sh

set -e

BACKUP_FILE=$1
TEST_DB="glide_hims_test_restore"

echo "[$(date)] Verifying backup: $BACKUP_FILE"

# Create test database
psql -U postgres -c "DROP DATABASE IF EXISTS $TEST_DB;"
psql -U postgres -c "CREATE DATABASE $TEST_DB;"

# Restore to test database
if [[ $BACKUP_FILE == *.gz ]]; then
    gunzip -c $BACKUP_FILE | pg_restore -U postgres -d $TEST_DB
else
    pg_restore -U postgres -d $TEST_DB $BACKUP_FILE
fi

# Run verification queries
echo "Verifying data integrity..."
PATIENT_COUNT=$(psql -U postgres -d $TEST_DB -t -c "SELECT COUNT(*) FROM patients;")
APPOINTMENT_COUNT=$(psql -U postgres -d $TEST_DB -t -c "SELECT COUNT(*) FROM appointments;")

echo "Patients: $PATIENT_COUNT"
echo "Appointments: $APPOINTMENT_COUNT"

# Check for recent data
LATEST_RECORD=$(psql -U postgres -d $TEST_DB -t -c "SELECT MAX(created_at) FROM patients;")
echo "Latest patient record: $LATEST_RECORD"

# Cleanup
psql -U postgres -c "DROP DATABASE $TEST_DB;"

echo "[$(date)] Backup verification completed successfully"

Disaster Recovery Plan

Recovery Time Objectives (RTO)
Scenario Target RTO Recovery Method
Database corruption 1 hour Restore from latest backup
Server failure 2 hours Deploy to new server + restore
Data center outage 4 hours Failover to secondary region
Ransomware attack 4-8 hours Clean server + restore from off-site
Recovery Checklist
  1. Assess the situation and identify affected systems
  2. Notify stakeholders and users
  3. Provision new infrastructure (if needed)
  4. Restore database from latest verified backup
  5. Restore file uploads
  6. Restore application configuration
  7. Deploy and start application
  8. Verify system functionality
  9. Run data integrity checks
  10. Resume normal operations
  11. Document incident and update procedures

Best Practices

Do
  • Encrypt backups at rest and in transit
  • Store backups in multiple locations
  • Test restore procedures regularly
  • Monitor backup job success/failure
  • Document recovery procedures
  • Use versioned storage (S3 versioning)
  • Include backup verification in automation
Don't
  • Store backups only on the same server
  • Skip backup verification
  • Keep backups unencrypted
  • Ignore backup failure alerts
  • Assume backups work without testing
  • Delete old backups without retention policy
  • Share backup credentials widely
